# Round 034 - HistoricalRuntime v0.3 异步 Job 与 Agent 进度面板

## 背景

用户反馈当前 Godot / Web Runtime 的体验还没有达到“越玩越有趣”的智能涌现感，因此本轮不再横向开新 demo，而是继续往 HistoricalRuntime 的 Agent 化深处开发。

本轮目标是先把生成过程从“点按钮后等待结果”改成“玩家能看到一个历史异常任务正在被多个工作组推进”。

## 实现内容

### 后端

文件：

```text
artifacts/web_story_loop_demo/server.mjs
```

新增：

```text
POST /api/generate/start
GET /api/jobs/{jobId}
```

新增运行时能力：
- 内存 Job 状态：`queued`、`running`、`succeeded`、`failed`。
- 稳定 stage：`queued`、`story_weaving`、`history_review`、`briefing`、`artifact_generation`、`commit_review`、`complete`、`failed`。
- 双层事件：`events` 面向用户，`technicalEvents` 面向开发/审计。
- 模型请求超时：`MODEL_REQUEST_TIMEOUT_MS`，默认 `12000`。
- 模型超时或失败时转 fallback，占位分支仍进入完整审计、简报、档案和提交流程。

### 前端

文件：

```text
artifacts/web_story_loop_demo/index.html
artifacts/web_story_loop_demo/script.js
artifacts/web_story_loop_demo/style.css
```

新增 Agent Runtime 面板：
- 进度条展示固定阶段。
- 前台 copy 使用产品语言：`叙事分析组`、`历史审计频道`、`情报值班台`、`档案组`、`值班系统`。
- 技术细节折叠展示真实 Agent 名和事件。
- 玩家选择分支后，前端改走 `start -> poll -> commit node` 的异步流程。

### 测试

文件：

```text
artifacts/web_story_loop_demo/server.test.mjs
artifacts/web_story_loop_demo/frontend_contract.test.mjs
```

新增覆盖：
- async Job start/status contract。
- 前台事件不泄漏 `StoryWeaverAgent` / `HistoryGuardAgent` / `RuntimeOrchestrator`。
- technical events 保留真实 Agent metadata。
- 模型请求超时会转 fallback，不会让 Job 永久 running。
- 前端包含 progress panel、polling 和技术事件区域。

## 结论

v0.3 第一刀已经让 Web Runtime 从同步按钮生成升级为可观察的 Agent Job。它还不是完整的智能涌现，但已经具备下一步扩展 `HistoricalCase`、`CandidateNode`、commit/reject、RepairAgent 和更复杂任务图的承载层。

## 下一步

优先做 v0.3 第二刀：

1. 将当前兼容结构正式拆成 `HistoricalCase`、`PlayerAction`、`CandidateNode`、`CommitResult`。
2. 让 HistoryGuard 可以阻止 candidate 进入正式故事图，而不是只显示审计提示。
3. 增加 RepairAgent 或 rewrite pass，用于高风险分支的重写/降级。
4. 让前端区分“候选分支生成中”和“世界线已提交”。

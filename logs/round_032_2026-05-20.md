# Round 032 - HistoricalRuntime v0.3 Agent Runtime spec

## Goal

Freeze the Agent-first architecture plan before v0.3 implementation.

## Confirmed decisions

- Agent is not a story character or chatbot window. The minimum unit is an auditable `AgentRun`.
- HistoricalRuntime is an event-driven Agent orchestrator.
- Player actions trigger the runtime, but the flow is determined by World State and Agent Task Graph.
- v0.3 uses a fixed serial task graph first: StoryWeaverAgent -> HistoryGuardAgent -> IntelDeskAgent -> ArtifactAgent -> Runtime commit.
- Agent communication uses structured intermediate objects, not shared raw chat context.
- HistoryGuardAgent has gatekeeping authority and can prevent a CandidateNode from entering the committed StoryGraph.
- StoryWeaverAgent creates CandidateNode only; Runtime commits to StoryNode after review and aggregation.
- MiniMax is a provider capability, not the source of this project's Agent semantics.
- Agent instances are task runs; long-term state lives in `HistoricalCase`.
- The growing object is the historical anomaly case/worldline itself, not a single character.
- Uncertainty is a world-growth mechanism, not a failure/punishment mechanism.
- v0.3 should migrate the `agent-progress-visibility-panel` component logic for real progress visibility.

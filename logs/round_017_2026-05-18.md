# Round 017 - 第一屏重点不是信息清单，而是信息呈现方式

日期：2026-05-18

## 本轮问题
围绕“1933 年爱因斯坦没有离开德国”，第一屏危机指挥室应该显示哪些具体信息？

## 用户纠偏
用户指出：比起呈现哪些具体信息，更应该关注这些信息的呈现方式。

也就是说，经过外部案例调研后，需要回答：

> 这些游戏中的信息呈现方式，哪些更适合本项目？怎样呈现才不会变成资料库、表格或百科？

## 本轮确认判断
第 17 轮重点从“信息内容清单”转向“信息呈现方式”。

第一屏不是把 Intel Desk、Timeline、Map、Archives、Risk Panel 的信息全部摊开，而是要设计一种渐进式、任务驱动、拟态内嵌的信息呈现。

## 从外部调研得到的呈现原则

### 1. 先给身份和压力，不先给全量系统
OS-like 游戏的第一分钟通常不是展示功能完整性，而是让玩家立刻知道：

- 我是谁？
- 我现在值什么班 / 接到什么任务？
- 为什么这件事紧急？

对本项目：第一屏应像“历史异常值班台”或“危机指挥室”，不是 App 启动器。

### 2. 信息以“任务流”呈现，而不是以“资料库”呈现
玩家不应先看到一堆可浏览材料，而应先收到一个待处理事件。

推荐链路：

```text
未读紧急情报
→ 情报中出现可疑矛盾
→ 系统给出弱建议
→ 玩家选择一个入口验证
→ OS 根据玩家动作逐步展开 Timeline / Map / Archives
```

### 3. 四个模块不应同时抢注意力
Timeline / Map / Archives / Intel Desk 可以同时存在，但第一分钟只有 Intel Desk 是主焦点。

其他模块应以“被情报点亮”的方式出现：

- Timeline 上出现一个闪烁异常节点。
- Map 上出现 2-3 个相关地点标记。
- Archives 显示“3 条相关证据待核验”。
- Intel Desk 显示当前情报、可信度、推荐操作。

### 4. 用拟态系统提示替代教程弹窗
不要写“点击这里学习时间线”。

更适合写：

- 值班规程提示
- 上级批注
- 系统审计提醒
- 来源可信度警告
- 档案一致性检查

这些提示既是引导，也是世界观的一部分。

### 5. 信息反馈要有“矛盾标记”，而不是百科解释
玩家要被驱动去问：

> 官方说法是真的吗？

因此呈现方式应突出矛盾：

- Baseline History vs Current Timeline
- Official Report vs Unverified Source
- Archive Present vs Archive Missing
- Map Trace vs Passport Record

## 对第一案的呈现建议

第一分钟可以这样组织：

```text
主焦点：Intel Desk 未读紧急情报
  - 表层：Subject Albert Einstein / Status: Still in Germany
  - 下层：真实历史基准中他本应离开德国
  - 矛盾：当前世界线显示他仍在柏林
  - 弱建议：在采取干预前，先确认情报来源与历史基准

周边被点亮：
  - Timeline：1933 年 3-4 月出现异常节点
  - Map：柏林、Caputh、安特卫普、普林斯顿出现地点标记
  - Archives：离境记录、科学院记录、住所搜查报道待核验
  - Risk：宣传系统、警察系统、国际救援网络出现风险标签
```

## 设计原则
第一屏应遵循：

> 不是“给玩家很多信息”，而是“让玩家看到一个正在崩开的历史矛盾”。

## 下一轮问题
第 18 轮：第一屏信息呈现应采用哪种视觉/交互范式？

候选：
1. 危机指挥室式：大屏态势 + 情报卡片 + 地图/时间线联动。
2. 桌面 OS 式：多个窗口、未读消息、档案夹、地图窗口。
3. 情报工作台式：中心案情卡 + 周围证据板、时间线、地图。
4. 档案审计式：像在审查一份异常档案，逐步展开矛盾。

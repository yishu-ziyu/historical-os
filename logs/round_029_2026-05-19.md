# Round 029 - 视觉方向可行性与 Godot 开发路径

## 触发

用户提供了新的视觉与技术路线判断：不要把项目做成角色在地图上跑来跑去的像素 RPG，而应优先做成“档案馆式架空历史模拟游戏”：纸上地图、素描人物、报纸/电报/年鉴 UI、少量像素图标。

## 子 Agent 结论

本轮派出三个子 Agent 分别探索：

- 档案素描风：最适合当前项目。资产压力低，能容纳大量人物和历史资料，适合 LLM 生成的档案、情报、报纸、年鉴和证据流。
- 极简像素战略地图风：适合作为地图标记、态势符号和低成本图标语言，但如果作为主屏会显得冷、薄、像数据面板，且容易削弱人物和档案感。
- 纸上兵棋 / 历史地图风：适合作为地图层和事件推演层，但不能把项目带偏成宏大战略兵棋；应服务于档案 OS，而不是取代主体验。

## 当前判断

三种方案不是平级候选。主方向是档案素描 / 情报档案 OS；纸上兵棋地图和极简像素风只作为局部视觉语言。

下一步如果进入 Godot，不应修改 Godot 引擎源码，而应新建具体游戏项目：

```text
Godot 4.x
2D
Compatibility renderer
GDScript
Control UI
JSON 数据驱动
```

## 用户纠偏

用户指出后两种风格 demo 看起来很拉，原因可能是整体渲染差。该判断成立：本轮快速 CSS demo 只能说明信息结构可行，不能作为视觉质量目标。后续评价美术方向时必须区分“风格可行性”和“当前占位渲染质量”。

## 对 Godot 工作流的说明

Godot 开发不是只在网页上做。通常是在 Godot Editor 里组织场景、Control UI、资源、脚本和导出设置；网页只是 Web 导出结果或快速验收预览。类似《杀戮尖塔》的卡牌/事件/状态结构可以用 Godot 做，因为核心是 UI、卡牌节点、状态机、数据表和回合逻辑；本项目对应的是档案卡、情报卡、事件卡、地图标记和历史状态更新。

# Round 023 - 核心转向：LLM 驱动的网状历史故事书

日期：2026-05-18

## 本轮用户核心表达
用户明确指出：当前想做的不是一款传统基础产品，也不是用固定代码直接决定下一个事件出现时间和内容。

真正想做的是：

> 用大语言模型 / AI 模型驱动的、基于历史框架约束的、具有随机性和自由生长能力的网状历史故事系统。

## 为什么要用 LLM
用户认为游戏要好玩，必须有随机性。

LLM 的输出具有随机性，但这种随机性不能完全失控，需要被某种框架选中和约束。

用户选择的约束框架是“历史框架”。

## 历史框架为什么有趣
历史有大量未被详细明史记录的小缝隙。

这些缝隙允许系统对细节做推演，例如：

> 如果爱因斯坦没有在纳粹占领德国前离开德国，他会发生什么？

可能分支：
- 他自杀了。
- 他被纳粹杀掉了。
- 他为纳粹德国效力。
- 他通过秘密渠道逃往美国。
- 他逃往法国或英国。

这些延伸不应该被完全预设，而应该在框架中自由生长。

## 多用户差异
即使三个用户选择了同一条发展路径，他们得到的结果也可以不同。

原因：
- LLM 输出具有随机性。
- 系统可以根据用户个人品味选择不同故事线。
- 同一历史节点可以长出不同网状分支。

## 核心价值标准
用户明确指出，这个系统首先是 for fun。

核心标准是：

1. 好玩。
2. 好看。

在这个基础上，再进一步讨论原则、原型和系统架构。

## 对当前 UI/UX 的判断
用户认为当前 UI/UX 应该大改。

新的方向更像：

1. 用户进入游戏。
2. 系统自动弹出一个历史事件。
3. 系统给出几个可能分支选项。
4. 用户也可以用自然语言输入自己的想法。
5. LLM 识别用户意图。
6. 故事开始自由发展。
7. 发展后生成几条后续故事线。
8. 用户选择想看的故事线继续跟进。
9. 用户可以在一定程度上回退。

## 新产品隐喻
用户提出新的隐喻：

> 一本网状的故事书。

或：

> 网状的历史。

它不是线性故事，也不是固定分支树，而是可生成、可回退、可继续探索的历史故事网络。

## 下一步问题
需要讨论：如果要实现这个产品，应该分哪些步骤？

初步方向：
- 先做“一个事件 → 选项/自由输入 → LLM 生成后续 → 生成新分支”的最小循环。
- UI 应该从复杂情报工作台转向更好玩、更好看的网状故事体验。
- 历史约束、用户品味、随机性和回退机制需要成为核心系统。
